## 📥 #001｜Authentication 完整拆解

**Source:** 知識庫熔爐 (@Know2learn_bot) forwarded PDF
**Origin:** YouTube `iX8g4LqF8p8` — 21min System Design Tutorial
**Topic tag:** `Phase-2: Architecture` / `Phase-3: Tech-Selection`
**Distill status:** ✅ READ
**SOP action:** → Phase 2 + Phase 3

### Raw notes（壓縮版）

**AuthN vs AuthZ 分清：**
- AuthN = Who are you? → 401
- AuthZ = What can you do? → 403
- 次序：AuthN → AuthZ

**7 種 Auth Methods（由舊到新）：**
1. Basic Auth（base64，必須 HTTPS）→ 內部工具
2. Digest（MD5，過時）→唔建議
3. API Key → 3rd-party / server-to-server，leak 永久
4. Session + Cookie（stateful，Redis 做 storage）→ 傳統 web apps
5. Bearer + JWT（stateless，scale 強）→ modern APIs
6. Access + Refresh Token（JWT pair，short-lived + long-lived）→ modern default
7. MFA → production 強烈建議

**JWT 係 token format，唔係 auth method。**
- JWT self-contained claims，唔使 DB lookup
- 配 access + refresh pair 係 modern default
- Refresh token 唔可以放 localStorage（XSS 風險），要用 HTTP-only cookies

**OAuth 2 = Authorization framework，唔係 authentication。**
- 答：呢個 app 可以 access 啲乜？
- 用於：第三方 app 授權（e.g. 你想第三方 app 讀你 Google Drive）

**OpenID Connect = OAuth 2 + Authentication。**
- 答：user 係邊個？
- 「Sign in with Google/GitHub/Microsoft」底層全部係 OIDC
- 同時返 access token + ID token

**SSO = UX pattern，唔係 auth method。**
- 底層靠：SAML（enterprise/legacy）或 OIDC（modern）

**Implementation stack recommendation：**
- Public website login → Username/password + MFA + Session cookie（HTTP-only + Secure + SameSite）
- API（internal/mobile）→ JWT access token（15min）+ Refresh token（HTTP-only cookie，7-30 days）
- 3rd-party integration → OAuth 2
- Enterprise SSO → SAML 2.0（legacy）或 OIDC（greenfield）

**Error codes：**
- 401 = 未 auth
- 403 = 已 auth 但無權限
- 400 = API key 缺失

### SOP 對應 Phase
- **Phase 2（架構）**：Stateless vs Stateful 選擇；Scaling 考慮
- **Phase 3（語言/框架）**：Auth stack 選擇決策矩陣

## 📥 #002｜RAG × 私有資料庫 系統設計

**Source:** 知識庫熔爐 (@Know2learn_bot) forwarded PDF
**Origin:** YouTube `NQZqET-jjws` — 14:34 System Design
**Topic tag:** `Phase-2: Architecture` / `Phase-2: Data`
**Distill status:** ✅ READ
**SOP action:** → Phase 2（數據架構 + 複雜度評估）

### Raw notes（壓縮版）

**Production-grade RAG 本質：**
- 10M+ documents、duplicate/conflicting/outdated、strict security、tight p95 latency
- 3 個核心 insight：
  1. **Data 處理係 bottleneck** — 90% effort 放 ingestion/dedup/chunking，唔係 fancy prompt
  2. **Retrieval 唔止係 similarity search** — query understanding + re-rank + hybrid search
  3. **Generation 唔可以 hallucinate** — grounding + cite source + cost control

**9 大子系統：**
1. Data ingestion（raw → ingest, deduplicate, chunk）
2. Pre-processing（chunking + metadata extraction）
3. Embedding（vector encoding, batch + GPU）
4. Vector DB（index storage + retrieval）
5. Query understanding（intent / reformulate / decompose）
6. Retriever（hybrid search + re-rank）
7. Prompt builder（compose context + system prompt）
8. LLM generator（inference + streaming + cache）
9. Post-processing（cite source + safety filter + log）

**Note:** 9 個系統唔係 sequential pipeline，係 **mesh**。

**Data 處理關鍵：**
- Exact dedupe (SHA-256)：100% 準，慢
- Fuzzy dedupe (MinHash)：95%+ 準，快，production common
- Chunking：parent-child / sentence-aware，唔係 fixed-size
- Metadata：author / date / language / topic，做 filterable

**Retrieval design：**
- BM25 / keyword：high precision for exact term
- Dense vector：high recall for semantic match
- Sparse (SPLADE / BGE-M3)：hybrid signal
- Re-rank：cross-encoder / Cohere Rerank，top-100 → top-5 先入 LLM context

**Generation 要點：**
- Citation：答案要 cite source document ID + page
- Refusal：冇足夠 context 時 refuse，唔好 hallucinate
- Streaming：TTFT (time-to-first-token) budget 控制
- Cost：cache hit 慳 tokens

**Security / Compliance：**
- PII detection + redaction
- Access control：per-user document visibility
- Audit log：邊個 user 問咩、system 答咩，全部 log 去 immutable storage
- Rate limit + abuse detection

**監控 5 大 metric：**
1. Latency p95
2. Retrieval recall
3. Generation faithfulness
4. Refusal 率
5. Cost-per-query

**5 個 production-grade mindset：**
1. Data 處理決定一切（90% effort 放 ingestion）
2. Hybrid retrieval + re-rank（純 vector唔夠）
3. Grounded generation（答案必須 cite source）
4. Security / Compliance by default（PII + ACL + audit一早 design 入）
5. Monitor everything

### SOP 對應 Phase
- **Phase 2（架構）：** 數據處理係 bottleneck呢個 insight、去重方法選擇、9-sub-system mesh vs pipeline
- **Phase 2（複雜度）：** 大規模數據（10M+）嘅特殊考慮

## 📥 #003｜Scalable File Upload Service 系統設計

**Source:** 知識庫熔爐 (@Know2learn_bot) forwarded transcript
**Origin:** YouTube `ka_WCmNdybE` — File Upload Service System Design
**Topic tag:** `Phase-2: Architecture` / `Phase-2: Data`
**Distill status:** ✅ READ
**SOP action:** → Phase 2（數據架構 + 存儲決策）

### Raw notes（壓縮版）

**File Upload Service 演進：**

**V1 — 直接上傳到 API + Database（問題多）：**
- File 緩衝喺 memory → API 成為 bottleneck
- Database 存二進制文件 → page size explosion（file 大過 DB page size → 跨多頁）
- 多人同時上傳 → 立即爆

**V2 — 引入 Object Store（S3 / Azure Blob）：**
- 唔直接上傳去 API，用 **pre-signed URL** 流程：
  1. Client → API：「我想上傳」→ API 返回 pre-signed URL
  2. Client → 直接上傳去 Object Store（帶 pre-signed URL）
- Object Store 優點：
  - Multipart upload（大文件拆細上傳）
  - Pause & resume
  - 内建 deduplication
  - File retention control
  - S3-compatible（自己 hosting 得，用 R2 / SeaweedFS / MinIO）

**V3 — Async Processing（Message Queue）：**
- 上傳完成 → Object Store 觸發 webhook / callback → API → Queue
- Queue consumer 做 background work：
  - Virus scanning
  - Thumbnail / preview generation
  - OCR
  - Validation
- 注意：Object Store 直接 queue message 比 API 代傳更 scalable

**V4 — CDN（公開/靜態文件）：**
- CDN 分發 public files（文檔、static assets）
- 全球分布 → 用戶就近 access
- 仍用 pre-signed URL 做 access control

**完整架構組件：**
```
Client
  ├── Pre-signed URL upload ──→ Object Store（S3/R2/Azure Blob/SeaweedFS）
  └── Pre-signed URL download ←─┘
                ↓ webhook/callback
         File Upload API（metadata + access control）
                ↓ message
              Queue（RabbitMQ/SQS）
                ↓ consumer
         Background Workers（virus scan / thumbnail / OCR）
                ↓
         CDN（public files）
```

**Self-hosted Object Store 選項：**
- ~~MinIO~~（已轉 closed source）
- Cloudflare R2（AWS S3 compatible， egress free）
- SeaweedFS（open source，S3 compatible）
- Rust-based S3-compatible options

**Storage 決策原則：**
- **大文件 / Binary** → Object Store，唔好放 Database
- **Metadata** → Database（SQL/RDS）
- **Static / Public files** → CDN
- **敏感文件** → Object Store + pre-signed URL，唔經過 API

### SOP 對應 Phase
- **Phase 2（數據架構）：** Binary files 唔放 DB；Object Store 係專門解决方案
- **Phase 2（架構）：** Pre-signed URL pattern；Async processing via queue

## 📥 #004｜J-Star Multi-Agent Framework

**Source:** 知識庫熔爐 (@Know2learn_bot) forwarded PDF
**Origin:** YouTube `HAoUVWO7ers` — 71min（復旦 AI 社區）
**Topic tag:** `Phase-2: Architecture` / `Phase-2: Agent` / `Phase-2: Workflow`
**Distill status:** ✅ READ
**SOP action:** → Phase 2（Agent Architecture）or Phase 2（Workflow）

### Raw notes（壓縮版）

**J-Star 係乜？**
- 10 萬 stars 開源 multi-agent 框架
- 重點唔係 deployment，係 **multi-agent 合作心法 + 多人專家協作 workflow**
- YC 總裁將 Cloud Code 工作流實戰經驗抽象成 framework

**7 階段 Sprint Workflow：**

| Phase | 名稱 | 主要內容 |
|---|---|---|
| 1 | 思考 | 需求分析、problem framing |
| 2 | 規劃 | 6 位專家協作，工件交接鏈 |
| 3 | 開發 | 工程師 + SAMO 評審 |
| 4 | 審查 | 代碼 / 結構 / 測試 review |
| 5 | 測試 | 整合 / e2e |
| 6 | 部署 | Release / 監控 |
| 7 | 反饋 | Trace / 改進 / 下一輪 |

**Phase 2 — 6 位專家協作（核心）：**

| 角色 | 職責 | 關鍵技巧 |
|---|---|---|
| Officer Outro | 結構化質問，逼定產品框架 | 6 類問題 |
| Chief Product Owner | 產品功能篩選 | 4 種模式：擴張/選擇性/扣張/保持 |
| Product Design Documenter | 撰寫設計文檔 | 文檔結構化 |
| Engineering Lead | 技術可行性 / 架構選擇 | SAMO 評估 |
| Dev Lead | 代碼結構 / module 設計 | Dev 評估 |
| Final Product Designer | 打磨完整思路 | 深度聚合 |

**3 條 Agent 開發核心原則：**

1. **角色級約束（Role-Level Constraint）**
   - 唔係寫 soft prompt（『請你做 investigate』）
   - 而係寫 **strict 硬約束**（step-by-step enumeration + black/white list）
   - Soft prompt = 建議性，Agent 容易 ignore
   - Strict constraint = 強制性，較少 hallucination

2. **把狐燒開（Disentangle）**
   - Agent 間通訊要 **結構化**、**可審計**
   - 中間狀態 / 結果 / trace 全部 **落文件系統**
   - 唔可以做黑盒

3. **Trace + Review**
   - 每次執行後做 **trace log**
   - 再 **review 改進約束**
   - Prototype 質量先會喺 single-turn 之上 **持續迭代**

**3 大底層原則（多 Agent 系統）：**

| 原則 | 意思 | 應用場景 |
|---|---|---|
| **Crowding** | 雙約束策略（Forward + Reverse） | 防止 Agent loop / 過度發散 |
| **Tight Constraints** | 絕色級約束，特定輸出鎖死 | 敏感任務 / 結構化輸出 |
| **Disentangle** | Output 可追溯，術後 review 可迭代 | Production-grade system |

**Soft Prompt vs Strict Constraint：**

| 寫法 | 性質 | 失敗 mode |
|---|---|---|
| Soft prompt（『請你做 investigate』）| 建議性 | Agent 容易 ignore，hallucination 累積 |
| Strict constraint（step-by-step + black/white list）| 強制性 | 較少 hallucination，但要更多寫約束時間 |

**5 個 Takeaway：**
1. **唔好被 deployment 工具層分散注意力** — 真正 value 喺 multi-agent 合作心法
2. **寫約束 ≠ 寫 prompt** — Prompt 係 suggestion，Constraint 係 enforcement
3. **多 Agent 唔係越多越好** — 規劃階段 6 位專家，其他階段 2-3 位即可
4. **Trace 係核心** — 每次跑落文件系統，後續 review 改進約束
5. **Open-source 框架係 starter** — Learn + fork + 改造，唔係從零自建

### SOP 對應 Phase
- **Phase 2（Agent Architecture）：** Multi-agent 協作模式、3 核心原則
- **Phase 2（Workflow）：** 7 階段 sprint、可套用於任何工作類型
